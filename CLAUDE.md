# CLAUDE.md

Guidance for Claude (and contributors) working in this Helm charts monorepo.

## Versioning & release automation — do not maintain changelogs by hand

Releases are fully automated. **Do not** hand-edit version fields, the
`artifacthub.io/changes` annotation, chart `CHANGELOG.md`, or release notes in a
PR, and **do not** flag them as stale or "not bumped" during review — they are
all (re)generated by CI after merge.

What is managed automatically:

- **`appVersion`** in each `charts/*/Chart.yaml` — bumped by **Renovate** when a
  new application image is released.
- **Chart `version`** — bumped by **Commitizen** in
  `.github/workflows/bump-version.yml` on push to `main`. Conventional commits
  drive the increment; Renovate/`chore` commits with no bump-worthy change get a
  forced PATCH bump. Per-chart config lives in `charts/<chart>/.cz.toml`.
- **`annotations.artifacthub.io/changes`** — regenerated from conventional
  commits since the chart's last tag by
  `scripts/update-artifacthub-changes.sh`, then committed as part of the
  Commitizen bump. This is what ArtifactHub shows as the chart changelog.
- **GitHub Release notes** (what Renovate surfaces in downstream consumer PRs) —
  `scripts/render-release-notes.sh` renders the `artifacthub.io/changes`
  annotation into a per-chart `release-notes.md` during
  `.github/workflows/helm-release.yml`, and `cr-config.yaml`
  (`release-notes-file: release-notes.md`) tells chart-releaser to use it as the
  Release body. Without this, chart-releaser falls back to the chart
  `description` and no changelog appears downstream.

### Release flow

`bump-version.yml` (push to `main`) detects changed charts → updates the
annotation → `cz bump` (commits version + annotation + `CHANGELOG.md`, creates a
`<chart>-<version>` tag) → calls `helm-release.yml`, which generates
`release-notes.md` per chart and runs chart-releaser to publish the packaged
chart and a GitHub Release.

### When you DO touch these

Only change the generators themselves (the scripts, `.cz.toml`, the workflows,
`cr-config.yaml`) — never the generated output in a feature PR. A normal feature
or fix PR should contain only `templates/`, `values.yaml`, docs, etc.; the
version/changelog churn lands on `main` afterward.

## Conventions

- Commits follow Conventional Commits; the scope is optional (e.g.
  `feat(helm): ...` or `feat: ...`). Both are picked up by the changelog
  tooling.
- Process YAML with `yq` (mikefarah/go-yq); see the scripts for examples.
- Do **not** add `*.md` or `release-notes.md` to a chart's `.helmignore`.
  chart-releaser reads the bundled `release-notes.md` (generated by
  `scripts/render-release-notes.sh`) to populate the GitHub Release body; if it
  is stripped from the package, `cr` silently falls back to the chart
  `description` and the downstream changelog regresses.
